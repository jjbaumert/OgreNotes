# OgreNotes Internationalization

Phase 5 M-P2 architectural doc. Locked-in design before the
harness lands — every later M-P2 piece (library integration, RTL
CSS, string-extraction pass, locale switcher) flows from this.

## Goals

OgreNotes is English-only as of Phase 4. Phase 5 M-P2 puts the
translation harness in place so that:

- **Every new UI string written after M-P2 is translatable from
  day one.** Strings authored before the harness ship in en-US;
  they get backfilled as the surfaces they live on are touched.
- **RTL languages render correctly** — at minimum Arabic-style
  layout flipping where logical CSS properties already make this
  cheap.
- **Locale-aware formatting** — dates, numbers, relative times
  follow the user's locale conventions rather than US-only.
- **Users pick their language explicitly** in profile settings
  (Phase 4 M-E2 admin console). Fallback chain: URL `?locale=`
  query → user stored pref → `navigator.language` → en-US.

Out of scope for Phase 5: translating into a long tail of
locales, in-app translation contribution UI, backend-string i18n
(emails, error messages). en-US + one RTL pilot (Arabic) is the
shipping target; additional locales are community-contributed or
land in a later milestone.

## Library choice — `fluent-rs`

`fluent-rs` (Mozilla Fluent) is the right choice. Decision:

| Candidate | Verdict | Rationale |
|---|---|---|
| **fluent-rs** | ✅ chosen | Pure Rust, wasm-compatible, runtime locale switching, rich syntax (plurals, gender, variants), used by Firefox. Active maintenance. |
| rust-i18n | rejected | Compile-time only. Switching locales at runtime requires a rebuild. Disqualifies the locale-switcher UX. |
| gettext-rs | rejected | C dependency. Adds build complexity to the wasm pipeline and the integration tests. |
| icu_messageformat | considered | Unicode-standard message format. Mature spec but heavier API surface; we don't need the extra power for v1. |
| t-rs | rejected | Newer / less battle-tested; smaller community. Worth a re-evaluation in v2 if it matures. |

### Fluent syntax cheat-sheet

```ftl
# Simple message
home-title = OgreNotes

# Interpolation
greeting = Hello, { $name }!

# Plurals via CLDR categories
unread-count =
    { $count ->
        [one] You have one unread message.
       *[other] You have { $count } unread messages.
    }

# Attributes (for the same message in different contexts)
login-button = Sign in
    .aria-label = Sign in to your account
    .title = Press Enter to sign in
```

Plurals + attributes + interpolation cover ~95% of real-world UI
strings. The full Fluent reference: <https://projectfluent.org/>.

## Architecture

### File layout

```
frontend/locales/
  en-US/
    main.ftl       — shared shell strings (sidebar, dialogs, errors)
    editor.ftl     — editor surface strings
    spreadsheet.ftl — spreadsheet surface strings (v2 split)
  ar/
    main.ftl       — Arabic translations (RTL pilot)
    editor.ftl
```

One `.ftl` file per locale per *domain* (logical bundle). A domain
is a chunk of related strings — keeping them split means a
translator can work on the spreadsheet surface without touching
the editor surface, and lazy-loading per domain is possible in v2.

For Phase 5 v1, ship just `main.ftl` — single domain, all UI
strings inside. The split into editor/spreadsheet domains is a
v2 carry-forward when the catalog grows past ~500 entries.

### Runtime resolution

A process-global `LocalizationManager` (singleton) holds:

- The currently-active locale (initialized once on app bootstrap)
- The loaded `FluentBundle` for that locale
- An en-US fallback bundle (always loaded — used when the active
  bundle is missing a key)

Lookup via a `t!` macro:

```rust
let label = t!("home.empty-folder");
let greeting = t!("home.greeting", name = user.name);
let count_msg = t!("notifications.unread-count", count = 5);
```

The macro expands to a method call on the manager that:
1. Looks up the message in the active bundle.
2. Falls back to en-US if missing.
3. Returns the message id itself (as a debug-visible string) if
   even en-US lacks it — never panics, never returns empty.

### Locale selection

Precedence, top wins:

1. **URL `?locale=ar`** — for testing + sharing localized URLs.
   Doesn't persist; resets on next page load.
2. **Server-stored pref (auth-response hint)** — `uiPrefs.locale`,
   carried on the `/auth/refresh` response body.
3. **`localStorage["ogrenotes.locale"]`** — warm cache of the
   user's last pick, written by `set_locale` when the switcher
   fires; the layer that keeps a device in the right locale on a
   logged-out or hint-less boot.
4. **`navigator.language`** — browser default.
5. **`en-US`** — last-resort fallback.

Applied at bootstrap inside `main.rs`, AFTER the boot refresh
resolves: the stored pref rides the `/auth/refresh` response as
`uiPrefs.locale` (no separate `/users/me` fetch), and
`resolve_locale_with_hint` folds it into tier 2 of the precedence
chain. Because init runs after the (already-awaited) refresh and
before mount, first paint is in the correct locale with no reload.
A logged-out boot (refresh 401) has no hint and falls through to
localStorage → navigator → en-US. In-session locale changes via the
switcher still call `set_locale` + reload (the reactive no-reload
re-render remains a v2 carry-forward).

### Bundle loading

Locale `.ftl` files are bundled into the WASM at compile time via
`include_str!`. v1 keeps both en-US and Arabic resident in memory
at all times (small footprint, ~10 KB per locale for the initial
catalog). v2 can introduce lazy-loading via `trunk`'s asset
pipeline + `fetch` if the catalog grows.

Compile-time inclusion has two benefits over runtime fetch:
- No network round-trip on locale switch — instant.
- No race condition between locale change and bundle-arrival.

The trade-off: every locale we ship adds to the WASM bundle.
v1 has two locales; if we add 10 more for the long tail, the
size will matter and we'll switch to lazy-load.

### RTL strategy

Two pieces:

1. **`<html dir="rtl">`** when the active locale is RTL. The
   manager flips this attribute alongside the locale. All
   downstream layout decisions inherit from `dir`.

2. **CSS logical properties everywhere.** Replace every
   `margin-left` / `padding-right` / `left:` etc. with
   `margin-inline-start` / `padding-inline-end` / `inset-inline-
   start:`. Browsers automatically swap physical edges based on
   `dir`. One CSS rule serves both LTR and RTL.

The audit pass (M-P2 piece 4) replaced every inline-direction
physical property in `frontend/style/*.css` with its logical
equivalent. Symmetric pairs (`left: 0; right: 0`) collapsed to
`inset-inline: 0`; full-bleed `top/left/right/bottom: 0` quartets
collapsed to `inset: 0`.

#### Patterns the audit settled on

- **Layout edges that should flip:** `margin/padding/border/inset
  -inline-(start|end)`, `text-align: start|end`.
- **Symmetric pairs:** `inset-inline: <value>`, `padding-inline:
  <value>` (one declaration replaces two).
- **Full-bleed positioning:** `inset: 0`.
- **`text-align`** on UI chrome (table headers, menu rows) →
  `start`. `text-align` written into **document content** (cell
  attrs, exported HTML for tables / images) stays physical
  because it represents the author's choice, not the UI direction.

#### Exceptions and v1 carry-forwards

- **Spreadsheet grid is pinned LTR.** `.spreadsheet-wrapper` sets
  `direction: ltr` regardless of `<html dir>`. Reasons (full
  rationale lives next to the rule in `spreadsheet.css`): column
  semantics A1 → B1 → C1 are universally LTR even in localized
  Excel/Numbers; the sticky row-header gutter + resize handles +
  paste-overlay + cell-corner-handle were each built against
  physical edges; formula syntax stays LTR (`=SUM(A1:B2)`) and
  mixing an RTL grid with LTR formulas creates a worse bidi mess
  than a consistent LTR grid embedded in an RTL page. Full RTL
  grid support is a v2 carry-forward.

- **Off-canvas drawers (sidebar, outline, conversation pane).**
  Drawer slide-in animations use `transform: translateX(...)`,
  which is NOT a logical property — it moves in physical pixels
  regardless of `dir`. The audit drove the translate sign through
  a CSS custom property:

  ```css
  .sidebar {
    --drawer-hide-x: -100%;
    transform: translateX(var(--drawer-hide-x));
  }
  [dir="rtl"] .sidebar { --drawer-hide-x: 100%; }
  .sidebar.is-open { transform: translateX(0); }
  ```

  Use the same pattern when adding any new slide-in/out
  animation that crosses an inline edge.

- **Remote-cursor caret + label.** `.remote-cursor-caret`
  (`border-left: 2px`) and `.remote-cursor-label` (`left: -1px`)
  stay physical for v1. The caret position is JS-computed from
  CRDT coordinates; full RTL caret rendering (which would need
  to flip the caret stripe edge based on the *surrounding text*
  direction, not page direction) is a v2 carry-forward in the
  CRDT / cursor-rendering layer.

- **Document-content alignment** (clipboard.rs / markdown.rs
  `text-align: left|right` test strings, cell-alignment exports)
  stays physical. Those values represent author intent within
  the document; they should round-trip unchanged regardless of
  whose UI loaded the document.

### Locale-aware formatting

Dates, numbers, and relative times don't go through `fluent-rs`
— they use the browser's native **`Intl`** API via the
`js_sys::Intl::*` extern bindings in `frontend/src/i18n.rs`.

```rust
use crate::i18n::{format_date, format_number, format_relative, DateStyle};

format_date(timestamp_usec, DateStyle::Medium)
//   en-US: "May 19, 2026"
//   ar:    "١٩ مايو ٢٠٢٦"

format_number(1_234_567.0)
//   en-US: "1,234,567"
//   de:    "1.234.567"
//   ar:    "١٬٢٣٤٬٥٦٧"

format_relative(timestamp_2_hours_ago_usec)
//   en-US: "2 hours ago"
//   ar:    "قبل ساعتين"
```

#### Why `Intl` instead of the `icu` crate family

Originally this section called for the `icu` Rust crate family.
M-P2 piece 5 switched to `Intl` for three reasons that compound:

1. **Bundle cost.** Every modern browser ships a full
   CLDR-backed ICU stack natively as the `Intl` JavaScript
   object. Calling it via wasm-bindgen costs **zero WASM
   bytes**. The equivalent `icu_datetime` + `icu_decimal` +
   `icu_relativetime` + their CLDR data crates land in the
   hundreds of kilobytes range (locale-data-baked-in flavor)
   and remain meaningful even after gzip — a budget Phase 5
   explicitly tracks (M-P9 sets a 850 KB gzipped WASM ceiling).

2. **Locale coverage.** Intl has every CLDR locale the user's
   browser ships, which on every shipping desktop + mobile
   browser since 2020 is the full set. The `icu` crates ship
   only the locales we bake in at compile time; adding a new
   locale means a code change and a release. With Intl, adding
   a locale is just a new `.ftl` file.

3. **`Intl.RelativeTimeFormat` quality.** Browsers' relative-time
   strings have been polished against every major locale for
   years. Hand-tuning equivalents per-locale via the `icu` crate
   is real work; piggybacking on the browser's investment is
   leverage.

`Intl.RelativeTimeFormat` has been Baseline-supported in every
shipping browser since 2020 — no polyfill needed. The handful of
extern bindings live in `frontend/src/i18n.rs`; js_sys already
binds `Intl.NumberFormat` and `Intl.DateTimeFormat`, but exposes
`.format` as a getter returning a bound JS Function (spec-accurate
but awkward at call sites), so the extern bindings re-bind the
two to flatten `.format` to a direct method call.

#### Relative-time tiered presentation

`format_relative` switches presentation by age, matching
activity-feed UX in every editor the codebase targets:

| Age | Format | Example (en-US) |
|---|---|---|
| < 60s | RelativeTimeFormat "now" | "now" |
| < 1h | RelativeTimeFormat minutes | "5 minutes ago" |
| < 1d | RelativeTimeFormat hours | "2 hours ago" |
| < 1w | DateTimeFormat weekday short | "Mon" |
| ≥ 1w | DateTimeFormat medium | "May 19, 2026" |
| future / clock-skew | DateTimeFormat medium | "May 19, 2026" |

Negative diffs (clock skew, future-dated timestamps) bypass the
"in N minutes" RelativeTimeFormat output — surfacing an absolute
date avoids presenting server-clock disagreement as a prediction.

## Build pipeline

`.ftl` files live under `frontend/locales/` (not `frontend/src/`)
so trunk doesn't try to treat them as Rust modules. They're
included via `include_str!` from `frontend/src/i18n.rs` at build
time.

The CI workflow already builds `frontend/` via `trunk build`; no
new step required.

## Lint + tooling

Two pieces of tooling that catch regressions during the rolling
string-extraction pass:

1. **`scripts/i18n-audit.sh`** (new) — grep for raw English
   strings inside `view!` macros that aren't wrapped in `t!()`.
   Flag-only initially; promote to a CI block-merge gate once
   the bulk-conversion pass is done.

2. **Missing-key detection** — at debug build time, the manager
   walks the en-US bundle on startup and warns about referenced
   keys that don't exist. Catches typos like `t!("home.titel")`
   before they ship.

## Locale switcher UX

Lives in the profile settings dropdown (the same surface as the
M-P1 theme selector). For Phase 5 v1, a simple `<select>` with
the two shipped locales. Future expansion:

- "System" option that tracks `navigator.language` (mirrors the
  theme selector's System mode).
- Per-document locale override (some teams keep one workspace in
  English even when the rest of the UI is localized).

## Backend strings

**Out of scope for Phase 5.** Server-generated email subjects,
notification text, validation error messages all stay English in
v1. The argument for deferring:

- Backend translation requires a parallel `.ftl` pipeline on the
  Rust server side (also doable with fluent-rs, but more setup).
- The user-facing volume is much smaller — most messaging happens
  through frontend-rendered surfaces.
- Backend strings often need locale context the request doesn't
  carry yet (e.g. "what locale was the user in when they signed
  up for this email").

A v2 milestone will tackle backend i18n. The frontend harness
landed here is the foundation — same Fluent + same `t!()` macro
shape — so the v2 work is mechanical replication, not a new
architecture.

## v2 carry-forwards

- **Additional locales beyond en-US + Arabic.** Each new locale is
  an additional `.ftl` file under `frontend/locales/<bcp47>/`.
  Translation contribution can be:
  (a) hand-edited PRs from community contributors,
  (b) an in-app translation editor (full v2 effort),
  (c) integration with a TMS (Crowdin, Lokalise) — out of scope
      until there's translator demand.
- **In-app translation editor.** Lets non-developers edit `.ftl`
  strings directly. Significant UX work; defer.
- **Backend i18n** (emails, error messages). Detailed above.
- **Locale-aware sorting and collation.** For doc-list ordering
  in non-English alphabets. icu_collator can do this; defer until
  someone reports the problem.
- **Per-document locale override.** Useful when a user wants to
  edit a doc in a language different from their UI locale.
  Stored on `DocumentMeta`, read by the editor surface. Defer.
- **Lazy-loaded locale bundles.** Once the catalog grows past
  ~500 entries per locale, switch to fetch-on-demand to keep the
  WASM bundle small.
- **Bidi text isolation** (`<bdi>` / `dir="auto"`) inside doc
  content. RTL within a primarily-LTR document needs careful
  isolation to avoid weird character reordering. Defer.
